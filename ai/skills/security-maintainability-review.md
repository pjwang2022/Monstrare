# 安全性與可維護性審查

適用於高風險計畫、合併（merge）前，以及一般的程式碼變更 code review。

## 審查優先順序

1. 功能性錯誤與回歸問題。
2. 安全性錯誤。
3. 隱私外洩。
4. 身分驗證與權限瑕疵。
5. 資料驗證缺口。
6. 遷移（migration）與回滾（rollback）風險。
7. 架構偏移。
8. 重複程式碼與可維護性。
9. 缺失的測試或薄弱的驗證。

## 必要輸出

```text
發現的問題：
- 嚴重程度：
- 檔案/行號（若可取得）：
- 問題：
- 影響：
- 建議修法：

殘留風險：
核准建議：核准 | 要求修改 | 阻擋
```

## 可維護性檢查項目

- 遵循既有模式。
- UI 變更取用 `ai/context/design-system.md` 的 token 與元件，沒有硬寫的一次性色彩／間距／字級；新做的元件已登記回元件庫 inventory。
- 避免重複抽象。
- 除非規格要求，否則保持公開 API 穩定。
- 保持函式與元件聚焦、單一職責。
- 只在能釐清非顯而易見的決策時才加註解。
- 行為改變時同步更新文件或情境地圖。
- 保持 diff 小到可審查。
- 避免不相關的格式異動。
- 優先選擇「無聊但可測試」的程式碼，而非炫技的寫法。

使用 `ai/checklists/security-checklist.md`、`ai/checklists/testing-checklist.md`，並參照 `ai/process/review-gates.md`。
