# 專案啟動（Project Kickoff）

適用於全新專案，或既有專案要一次性建立完整的 Epic → User Story → Task 待辦清單時。目的是把「這個專案要做什麼」逐層拆解成看板上的任務卡；每一層都要停下來讓人工勾選確認，不能自己悶頭把三層全部生完才回報。

## 步驟

### 1. 專案規格釐清

在提出任何功能模組之前，先確認：

- 部署目標（例如：Vercel、自架 server、App Store、內部工具）。
- 技術棧（語言、框架、資料庫、認證方式）。
- 已知的硬性限制（預算、時程、既有系統整合）。

用最少的問題把這些問完；使用者已經講清楚的部分不要重問。

### 2. 專案設置（Epic 0），一次性、必須全部完成才能往下走

任何專案都要先有一個固定名稱的 Epic「專案設置」，寫進 `tools/kanban/epics.json`。**這個 Epic 不是候選項目，不得用 AskUserQuestion 或任何形式的勾選清單問使用者要不要做**——它是每個專案都必須完成的基礎工作，不是使用者的偏好選擇。用一段文字告知使用者「將建立以下 User Story」即可，不用等待勾選核准；使用者若主動要求增減，才依指示調整。

依步驟 1 已確認的技術棧／部署目標，決定這個 Epic 底下要包含哪些 User Story（不適用的類別就不建立，例如純後端專案不需要 UI 設計系統這個 User Story）：

- 技術骨架初始化（專案初始化、目錄結構、基礎 dependency、build/lint/test 管線、部署設定）。
- 環境變數與金鑰設定（`.env.example`、密鑰存放規範）。
- 核心資料模型／認證框架基礎（跨 Epic 共用的核心資料表或登入機制，若專案有的話）。
- UI 設計系統／風格指引（若專案有 UI 的話）。**這裡不是只挑色票、做一個元件範例就結束**：先依步驟 1 判斷專案有哪幾種主要介面／使用者端（例如管理員後台、顧客前台，各自算一個 User Story 或一張任務卡），針對每一端套用 `ui-mockup-gate` skill，產出 2-3 個完整版面（layout）的 mockup 變體——包含整體版型、資訊架構、導覽與內容區配置，不是單一元件或色盤——等使用者從每一端選定一個版型後，才把選中的版型歸納整理成共用的色彩、字體、間距、共用元件庫規範。

這些 User Story 選定後，跟其他 Epic 一樣，用 `implementation-plan` skill 依需求拆成有明確規格、範圍、驗收標準的任務卡（不是一個類別一張大卡）；若某個 User Story 範圍還不清楚，一樣可以先用 `spec-interrogation` skill 釐清。

這個 Epic 底下的任務卡：

- `epic` 一律填「專案設置」，`userStory` 對應到上面選定的 User Story 名稱。
- `order` 給比其他 Epic 的卡片都小的值，讓看板上永遠排最前面。
- `dependsOn` 一律是空陣列（它們是相依圖的根，其他任務卡會依賴它們，它們不依賴任何人）。

**強制規則**：在「專案設置」Epic 底下所有任務卡都變成 `done` 之前，其他任何功能 Epic 的任務卡都不能被推進到 `ready` 之後的階段。落實方式：往下每個功能 Epic 最先建立的那張任務卡（若該 Epic 套用「Epic 架構優先規則」，就是那張「{Epic} 架構基礎」卡；否則就是這個 Epic 底下最先拆出的任務卡），`dependsOn` 必須列上「專案設置」Epic 底下**全部**卡片的 id，不是「有用到才列」。

### 3. 提出功能模組（Epic）並讓使用者勾選

依照專案主題，主動發想一組合理的功能模組候選清單，不要空手叫使用者自己想。**先把完整候選清單寫下來、數一下總數，再套用下方「候選清單呈現規則」決定要問幾輪**，用多選問題（例如 AskUserQuestion）列出候選項目，讓使用者勾選要哪些、或補充遺漏的模組。

使用者勾選確認後，不能就地照候選清單的原始順序寫入，要先幫這些 Epic 排出建置順序：判斷每個 Epic 對其他 Epic 的影響範圍——越是核心資料模型、主要使用者旅程、其他模組都會依附的底層需求，排越前面；越是周邊、獨立、可有可無的小功能，排越後面。這個順序**不用再讓使用者勾選**，用一段文字簡短說明排序理由即可（例如「先建 A 因為 B、C 都會用到它的資料模型」），不用等待核准；使用者若覺得順序不對，才依指示調整。

選定的每個 Epic，依上面排好的順序寫進 `tools/kanban/epics.json` 的 `epics[]`（`name` / `definition`）；陣列順序就是看板「藍圖」分頁的顯示順序，也是步驟 6 決定先做哪個 Epic 的依據，Epic 0「專案設置」永遠排在最前面，不受這個排序規則影響。

### 4. 針對每個 Epic 提出使用者需求（User Story）並讓使用者勾選

對每一個已選定的 Epic，主動發想這個模組底下合理的使用者需求候選清單。**同樣先數過總數、套用下方「候選清單呈現規則」**，一樣用多選方式讓使用者勾選確認或增刪。

選定的每個 User Story，寫進對應 Epic 底下的 `stories[].name` / `definition`。

若某個 Epic 的範圍還不清楚，先用 `spec-interrogation` skill 產出這個 Epic 的 `ai/templates/feature-spec.md`；規格書裡的「User Stories」章節就對應這裡勾選出的清單。

### MECE 檢查（適用於步驟 3 與步驟 4，在呈現候選清單給使用者之前完成）

腦力激盪候選清單時，不能只憑直覺條列、也不能只從使用者已經明講的需求出發，要用 MECE（相互排斥、完全窮盡）當品質門檻，並以「市面上同類系統常見的標準功能模組／需求」為外部基準。

- **完全窮盡（CE）**：先判斷這個系統屬於哪一種常見類型（例如電商、SaaS 後台、預約系統、內容平台、社群、內部工具），在心裡列一份這類系統的「業界標準模組地圖」，再逐一核對目前的候選清單是否涵蓋。特別留意容易被漏掉的骨架型模組：認證與權限、通知、搜尋／篩選、稽核紀錄、後台管理、報表／儀表板、設定／偏好。缺的要主動補上再交給使用者勾選，不能因為使用者沒提到就跳過。
- **相互排斥（ME）**：候選之間不能有職責重疊或邊界模糊的情況。若兩個候選其實是同一件事的不同說法（例如「會員系統」與「登入系統」高度重疊），要先合併或講清楚各自邊界，才能拿去問使用者；不要把會混淆使用者判斷的重疊候選都丟出去問。
- User Story 層級同樣適用：同一個 Epic 底下的 User Story，要覆蓋這個模組的完整使用者旅程（例如建立／查看／編輯／刪除／搜尋、正常路徑與例外路徑），而不是只列一兩個最直覺的情境；彼此之間也不能是同一個情境的重複描述。

### 候選清單呈現規則（適用於步驟 3 與步驟 4，強制檢查，不可省略）

多選問題工具（例如 AskUserQuestion）**一題最多 4 個選項、一次呼叫最多 4 題**——等於一次呼叫最多攤開 16 個候選項，這是工具的硬性上限，無法在單次呼叫裡繞過。**不得因為這個介面限制，就把腦力激盪出的候選清單砍到只剩 4 個或 16 個**，這會讓使用者誤以為看到的就是全部、實際上卻漏看了本來可能更好的候選。

**呼叫 AskUserQuestion 之前，強制自問並回答這一題：「我剛剛腦力激盪出的候選總數是多少？」** 把這個數字明確寫出來，再決定輪數，不要沒算過總數就先丟一題 4 選項出去。

- 候選數 ≤ 4：一題問完。
- 4 < 候選數 ≤ 16：拆成多題（依主題分組，例如「核心模組／延伸模組」），在**同一次呼叫**裡一起問完（一次呼叫最多 4 題）。
- 候選數 > 16：**必須拆成多輪、多次呼叫 AskUserQuestion**，不能只呼叫一輪就當作問完了。每一輪呼叫前都要在文字訊息中明確告訴使用者「這是第 N 輪，總共約 M 個候選，還有更多」；跑完所有輪次、彙整完使用者在每一輪的勾選結果之後，才能進入下一步驟。
- 使用者永遠可以用「其他」欄位手動補充，但這只是補充機制，不能取代主動窮舉候選、主動分輪呈現的責任。
- 若某一題剛好 4 個選項全被選走，代表這個分類可能還有沒攤開的候選，必要時主動追加一輪確認是否有遺漏。
- **完成檢查**：在彙整 Epic／User Story 清單前，回頭確認「已呈現給使用者的候選總數」是否等於「腦力激盪出的候選總數」。若少於，代表漏問了幾輪，必須補問，不得直接視為完成。

### 5. 拆分任務（Task）並推上看板

對每個已選定的 User Story，用 `implementation-plan` skill 拆成任務卡，全端功能依該 skill 的「全端功能拆分規則」預設拆成前端／後端／前後端串接三張卡；若該 Epic 需要先立架構基礎，依該 skill 的「Epic 架構優先」規則先拆架構基礎卡。

每張任務卡都要建立成 `tools/kanban/` 底下一張真實卡片（一卡一個 JSON 檔，`stage` 從 `backlog` 開始），並設定 `epic` / `userStory` 對應到上面選定的名稱，讓看板的「藍圖」分頁能正確歸類、算出完成度。除了步驟 2 規定的「每個功能 Epic 第一張卡要 dependsOn 全部『專案設置』卡片」，同 Epic 內若套用了架構基礎卡，其餘卡也要在 `dependsOn` 填上該架構基礎卡的 id。`backlog` 車道內的 `order` 也要照步驟 3 排好的 Epic 順序給值，較前面（較底層／較基礎）Epic 的任務卡 `order` 要比較後面 Epic 的任務卡小，讓看板從上到下就是建議的建置順序。

### 6. 逐一任務的實作迴圈

每次只處理一張已在看板上、狀態為 `ready` 的任務卡。若同時有多張卡都是 `ready`，優先選 `epics.json` 中排序較前面（較底層／較基礎）Epic 底下的那張，同一個 Epic 內再依 `order` 由小到大處理，不要跳著挑周邊小功能先做：

1. 若是前端任務、且對應畫面還不存在，先跑 `ui-mockup-gate`，做 2-3 個 mockup 讓使用者選一個，選完才進入下一步；既有功能的調整直接跳過這步。
2. 探索是否有現成套件或框架可以解決這個任務，避免自己重造輪子。若有 2-3 個合理選項，列出來問使用者要用哪個並給出自己的建議；只有一個明顯選擇時才不用問。
3. 開始實作。每個階段性成果都要讓系統維持「完整可用」的狀態，不能留下寫一半、會讓系統壞掉或啟動不了的半成品。
4. 自己檢查正確性：寫並跑過單元測試與整合測試，不是只憑肉眼看程式碼。
5. 確認沒問題後，把看板上這張卡的狀態往前推（`implementing` → `verify` → `done`），並視情況記錄驗證證據（對應 `test-verification` skill 與 `ai/templates/verification-report.md`）。

### 7. 資安紀律（每一張任務卡都要做，不是只有高風險才做）

實作當下就要套用 `ai/checklists/security-checklist.md`，包含但不限於：

- 不得以明文儲存密碼或密鑰。
- 檢查 OWASP Top 10 相關風險。
- 高風險項目（認證、金流、個資、基礎設施）仍需額外走 `ai/process/review-gates.md` 的 Security Gate。

## 輸出

- 已確認的技術棧與部署目標。
- `tools/kanban/epics.json` 內已選定的 Epic 與 User Story，含固定排最前面的「專案設置」Epic，其餘 Epic 依基礎程度由大到小排序。
- `tools/kanban/cards/` 內對應的任務卡（含 `epic` / `userStory` / `dependsOn`）。
- 逐張任務卡的實作與驗證紀錄。
