# 實作計畫

在產品與 UI 關卡皆已核准後使用此流程。

## 步驟

1. 閱讀已核准的功能規格書與情境包。
2. 確定可能會變更的檔案。
3. 定義 API、資料與狀態（state）契約。
4. 指定風險等級。
5. 使用 `ai/templates/task-card.md` 將工作拆分為小型任務卡。
6. 確保每張任務卡都符合 `ai/process/definition-of-ready.md`。
7. 若風險為中或高，在實作前停下來等待核准。

## 全端功能拆分規則

若一個 User Story 同時涉及前端與後端，預設拆成三張任務卡，而不是一張大卡：

1. 前端（畫面、互動、狀態管理）。
2. 後端（API、資料模型、商業邏輯）。
3. 前後端串接（實際串接 API、處理載入／錯誤狀態、端到端驗證）。

這樣每張卡維持「一次只做一件事」，風險與驗證範圍也更容易掌握。純前端或純後端的工作不需要硬套這個三分法。

## 任務拆分的 MECE 檢查

拆完任務卡後，回頭核對，不可直接視為完成：

- **完全窮盡（CE）**：所有任務卡合起來，要能覆蓋這個 User Story 的完整驗收標準；若規格書裡有提到、卻沒有任何任務卡負責，代表有缺口，要補一張卡或修正既有任務卡的範圍。
- **相互排斥（ME）**：任務卡之間不能有重疊的職責（例如兩張卡都要改同一支 API 或同一個元件的核心邏輯）；每張卡的邊界要寫清楚，避免實作時界線不清、互踩。

## 輸出

- 架構筆記。
- 任務卡。
- 相依關係。
- 驗證計畫。
- 所需的審查關卡。
