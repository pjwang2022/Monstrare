# 實作計畫

在產品與 UI 關卡皆已核准後使用此流程。

## 步驟

1. 閱讀已核准的功能規格書與情境包。
2. 確定可能會變更的檔案。
3. 定義 API、資料與狀態（state）契約。
4. 指定風險等級。
5. 使用 `ai/templates/task-card.md` 將工作拆分為小型任務卡。
6. 確保每張任務卡都符合 `ai/process/definition-of-ready.md`。
7. 若風險為中或高，在實作前停下來等待核准。

## 全端功能拆分規則

若一個 User Story 同時涉及前端與後端，預設拆成三張任務卡，而不是一張大卡：

1. 前端（畫面、互動、狀態管理）。
2. 後端（API、資料模型、商業邏輯）。
3. 前後端串接（實際串接 API、處理載入／錯誤狀態、端到端驗證）。

這樣每張卡維持「一次只做一件事」，風險與驗證範圍也更容易掌握。純前端或純後端的工作不需要硬套這個三分法。

## Epic 架構優先規則

拆某個 Epic 底下的任務卡之前，先判斷：這個 Epic 底下的多個 User Story，是不是共用同一組畫面框架、路由保護，或資料模型？

- 若是，且這組共用基礎目前還沒有任何已完成（`stage: done`）的任務卡建立過，要先拆一張「{Epic 名稱} 架構基礎」任務卡，內容是把這組共用基礎（例如後台的 layout、權限中介層、核心資料表）一次做好；同一個 Epic 底下接下來建立的其他任務卡，都要在 `dependsOn` 陣列裡列上這張架構基礎卡的 id。
- 若某個 Epic 範圍單一、User Story 之間不共用架構或資料模型，不需要套用這條規則，避免小 Epic 被過度拆卡。
- 若某張任務卡用到 `project-kickoff` 步驟 2 建立的「基礎建設」Epic 底下的卡片（例如共用的認證框架、專案骨架），同樣要在 `dependsOn` 裡列上對應卡片 id。

## 任務拆分的 MECE 檢查

拆完任務卡後，回頭核對，不可直接視為完成：

- **完全窮盡（CE）**：所有任務卡合起來，要能覆蓋這個 User Story 的完整驗收標準；若規格書裡有提到、卻沒有任何任務卡負責，代表有缺口，要補一張卡或修正既有任務卡的範圍。
- **相互排斥（ME）**：任務卡之間不能有重疊的職責（例如兩張卡都要改同一支 API 或同一個元件的核心邏輯）；每張卡的邊界要寫清楚，避免實作時界線不清、互踩。

## 輸出

- 架構筆記。
- 任務卡。
- 相依關係（`ai/templates/task-card.md` 的「前置任務（dependsOn）」欄位，對應看板卡片的 `dependsOn`）。
- 驗證計畫。
- 所需的審查關卡。
