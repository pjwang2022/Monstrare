# 規格書質詢（Spec Interrogation）

在實作前使用此流程。適用範圍是單一功能模組（Epic）。若是全新專案、還沒有 Epic 清單，先用 `project-kickoff` 流程建立並讓人工勾選 Epic，再對每個 Epic 個別跑這個流程。

## 步驟

1. 將需求重述為「問題」而非「解法」。
2. 確定使用者、目標、非目標與風險。
3. 只詢問阻擋性（blocking）問題。若某假設合理且安全，則直接記錄下來。
4. 產出 `ai/templates/feature-spec.md`。
5. 使用可測試的需求語言撰寫。
6. 在進入架構規劃前，停下來等待人工核准。

## 必要章節

- 問題。
- 使用者。
- 目標。
- 非目標。
- 使用者故事清單（User Stories，每條可獨立驗收，之後會逐條拆成任務卡）。
- 使用者旅程。
- 功能需求。
- 若涉及 UI，列出相關畫面。
- 資料與 API 假設。
- 安全性與隱私備註。
- 驗收標準。
- 驗證計畫。
