# 審查關卡（Review Gates）

用關卡讓人工決策權在流程早期就介入。

## 產品關卡

檢查項目：

- 這個問題值得解決嗎？
- 非目標是否清楚？
- 使用者流程與驗收標準是否完整？
- 邊界案例是否涵蓋？

核准者：

- 人工產品負責人。

## UI Mockup 關卡

檢查項目：

- 是否有多個變體？
- 是否涵蓋預設、空狀態、載入中、錯誤與行動裝置版狀態？
- 變體是否以 `ai/context/design-system.md` 既有的 design token 與元件組成（Epic 0 的風格方向階段除外）？
- 新做的元件是否依既有風格製作、並已登記回元件庫 inventory？
- 選定的變體是否符合產品意圖？
- 視覺驗收標準是否清楚？

核准者：

- 人工產品負責人或設計師。

## 架構關卡

檢查項目：

- 計畫是否遵循既有模式？
- 是否避免不必要的抽象？
- 資料契約是否清楚？
- 遷移與回滾風險是否已被理解？

核准者：

- 架構 agent，高風險任務再加上人工審查。

## 安全性關卡

檢查項目：

- 輸入驗證。
- 身分驗證與授權。
- 密鑰處理。
- 注入風險。
- 檔案與網路邊界。
- 敏感資料的記錄行為。
- 依賴套件與供應鏈風險。

核准者：

- 安全性審查 agent，高風險任務需人工核准。

## 驗證關卡

檢查項目：

- 是否有針對預期行為的測試。
- 回歸風險是否已涵蓋。
- CI 相關指令是否通過。
- UI 螢幕截圖是否已審查（如適用）。

核准者：

- 測試工程 agent 或人工審查者。

## 合併關卡

檢查項目：

- Diff 夠小。
- 摘要清楚。
- 已附上證據。
- 審查發現已解決或已記錄。
- 已記錄人工驗收。
